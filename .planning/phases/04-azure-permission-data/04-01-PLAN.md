From js/data/aws.js (reference pattern):
```javascript
// Each feature object has: id, name, question, [subQuestion], actions[], rationale{}, terraform, setupGuide
// Multi-account is special: uses policies[] instead of actions[]
export const AWS_FEATURES = { vpcIpamDiscovery, ec2Networking, ... };
```

Azure equivalent structure:
```javascript
// Each feature has: id, name, question, [subQuestion], roles[] (built-in), [customRole] (JSON), terraform, setupGuide
// roles[]: { name, builtIn: true, scope }
// customRole: { name, permissions[], scope } — only for Cloud Forwarding
export const AZURE_FEATURES = { ipamAssetDiscovery, publicDnsReadOnly, ... };
```
</interfaces>

Create js/data/azure.js as an ES module following the same constant-per-feature pattern as aws.js. Azure uses a role-based model instead of IAM actions, so the data shape differs.

**Data shape per feature:**
Each feature object has: `id`, `name`, `question`, optional `subQuestion`, `roles` array (built-in roles), optional `customRole` object (for Cloud Forwarding), `terraform` template string, `setupGuide` template string, and `rationale` object.

**Feature 1: ipamAssetDiscovery (AZR-01)**
- id: 'ipamAssetDiscovery'
- name: 'IPAM / Asset Discovery'
- question: 'Discover VNets, subnets, VMs, IP addresses?'
- roles: [{ name: 'Reader', builtIn: true, scope: 'subscription' }]
- rationale: { 'Reader': 'Read-only access to all Azure resources for network topology and IP address discovery' }
- terraform: azurerm_role_assignment resource block assigning the Reader role to the service principal at subscription scope. Use `data "azurerm_subscription" "current" {}` for scope. Use `data "azurerm_role_definition" "reader"` with `name = "Reader"` and `scope = data.azurerm_subscription.current.id`.
- setupGuide: Steps to create service principal, assign Reader role at subscription scope via Azure Portal, configure credentials in Infoblox Portal.

**Feature 2: publicDnsReadOnly (AZR-02)**
- id: 'publicDnsReadOnly'
- name: 'Public DNS - Read-Only'
- question: 'Sync public DNS zones?'
- subQuestion: 'Read-only'
- roles: [{ name: 'Reader', builtIn: true, scope: 'subscription', note: 'Covered by Reader role — no additional assignment needed if IPAM/Asset Discovery is also selected' }]
- rationale: { 'Reader': 'Read-only access includes DNS zone enumeration and record listing' }
- terraform: Same azurerm_role_assignment for Reader (comment noting it is shared with IPAM). Include a comment: `# Reader role covers public DNS read-only — no additional assignment needed`
- setupGuide: Note that Reader role covers this; if already assigned for IPAM discovery, no additional step needed.

**Feature 3: publicDnsReadWrite (AZR-03)**
- id: 'publicDnsReadWrite'
- name: 'Public DNS - Read-Write'
- question: 'Sync public DNS zones?'
- subQuestion: 'Read-write (bidirectional)'
- roles: [{ name: 'Reader', builtIn: true, scope: 'subscription' }, { name: 'DNS Zone Contributor', builtIn: true, scope: 'subscription' }]
- rationale: { 'Reader': 'Read-only access to discover existing DNS zones and records', 'DNS Zone Contributor': 'Create, update, and delete DNS zones and records for bidirectional sync' }
- terraform: Two azurerm_role_assignment blocks — one for Reader, one for DNS Zone Contributor. Use `data "azurerm_role_definition" "dns_zone_contributor"` with `name = "DNS Zone Contributor"`.
- setupGuide: Steps to assign both Reader and DNS Zone Contributor at subscription scope.

**Feature 4: privateDns (AZR-04)**
- id: 'privateDns'
- name: 'Private DNS Sync'
- question: 'Sync private DNS zones?'
- roles: [{ name: 'Private DNS Zone Contributor', builtIn: true, scope: 'subscription' }]
- rationale: { 'Private DNS Zone Contributor': 'Manage private DNS zones and records for internal name resolution sync' }
- terraform: azurerm_role_assignment for Private DNS Zone Contributor. Use `data "azurerm_role_definition" "private_dns_zone_contributor"` with `name = "Private DNS Zone Contributor"`.
- setupGuide: Steps to assign Private DNS Zone Contributor at subscription scope.

**Feature 5: cloudForwardingDiscovery (AZR-05)**
- id: 'cloudForwardingDiscovery'
- name: 'Cloud Forwarding - Discovery Only'
- question: 'Manage Azure DNS Private Resolvers?'
- subQuestion: 'Discovery only'
- roles: [] (no built-in role covers this)
- customRole with name "Infoblox UDDI - DNS Resolver Discovery" and exactly these 6 permissions:
  1. Microsoft.Network/dnsResolvers/read
  2. Microsoft.Network/dnsResolvers/inboundEndpoints/read
  3. Microsoft.Network/dnsResolvers/outboundEndpoints/read
  4. Microsoft.Network/dnsForwardingRulesets/read
  5. Microsoft.Network/dnsForwardingRulesets/forwardingRules/read
  6. Microsoft.Network/dnsForwardingRulesets/virtualNetworkLinks/read
- rationale: One entry per permission explaining what it reads.
  - 'Microsoft.Network/dnsResolvers/read': 'List and read DNS Private Resolver instances'
  - 'Microsoft.Network/dnsResolvers/inboundEndpoints/read': 'Read inbound endpoint configurations'
  - 'Microsoft.Network/dnsResolvers/outboundEndpoints/read': 'Read outbound endpoint configurations'
  - 'Microsoft.Network/dnsForwardingRulesets/read': 'List and read DNS forwarding rulesets'
  - 'Microsoft.Network/dnsForwardingRulesets/forwardingRules/read': 'Read individual forwarding rules'
  - 'Microsoft.Network/dnsForwardingRulesets/virtualNetworkLinks/read': 'Read VNet links associated with forwarding rulesets'
- terraform: azurerm_role_definition resource creating the custom role at subscription scope, then azurerm_role_assignment to assign it. The role definition uses `permissions { actions = [...] not_actions = [] }`.
- setupGuide: Steps to create custom role via Azure Portal or CLI, then assign to service principal.
