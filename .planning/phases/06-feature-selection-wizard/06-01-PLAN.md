**js/questions.js** — Create a new ES module that derives wizard questions from feature data modules.

Export a single function `getQuestionsForProvider(providerId)` that returns an array of question objects:

```javascript
// Question object shape:
{
  id: string,              // unique question ID, e.g., 'aws-vpc-ipam'
  text: string,            // question text from feature data (e.g., 'Discover VPCs, subnets, IP address management?')
  featureIds: string[],    // feature IDs this question controls (single for simple, empty for parent-only)
  subQuestions: [           // only present if this question has sub-features
    {
      id: string,          // e.g., 'aws-dns-readonly'
      text: string,        // sub-question label from subQuestion field
      featureId: string,   // the specific feature ID to toggle
      exclusive: boolean   // true = radio-style (only one sub-option), false = can multi-select
    }
  ]
}
```

Grouping logic:
- Features with no `subQuestion` field and a unique `question` string = standalone question. `featureIds` contains the single feature ID.
- Features sharing the same `question` string = grouped under one parent question. Parent `featureIds` is empty (sub-questions handle selection). Each becomes a subQuestion entry.
- DNS sub-questions are exclusive (radio-style: pick read-only OR bidirectional, not both).
- Cloud Forwarding sub-questions: For AWS, exclusive (discovery OR full). For Azure, exclusive. For GCP, NOT exclusive (inbound + outbound can both be selected — they produce "both" when combined).
- Multi-account/subscription/project features are standalone questions (no subQuestion field).

Import the FEATURES objects:
```javascript
import { AWS_FEATURES } from './data/aws.js';
import { AZURE_FEATURES } from './data/azure.js';
import { GCP_FEATURES } from './data/gcp.js';
```

Build the question list by iterating over the feature objects for the given provider, grouping by `question` string, and creating the structure above. The order should match the object key order in the FEATURES export (which reflects logical category ordering).

**js/state.js** — Extend with three new exports:

1. `setFeature(providerId, featureId, enabled)` — Sets `state.providers[providerId].features[featureId] = enabled`. If `enabled` is false, delete the key. Returns the updated features object.

2. `getFeatures(providerId)` — Returns `state.providers[providerId].features` object (keys are feature IDs, values are `true`).

3. `setSelectionMode(mode)` — Sets `state.selectionMode = mode` where mode is `'wizard'` or `'advanced'`. Default is `'wizard'`.

4. `getSelectionMode()` — Returns current mode string.

Initialize `state.selectionMode = 'wizard'` in the state object.

The `hasProviderData` function already checks `Object.keys(features).length > 0` — this will naturally work with the new feature keys.

**js/ui.js** — Add a new exported function `renderWizard(questions, features, onAnswer)`:

- `questions` is the array from getQuestionsForProvider
- `features` is the current features object from state
- `onAnswer` is a callback: `(featureId, enabled) => void`

The function renders into `#workspace-content`:

1. For each question, create a question card div:
   - Active question (first unanswered): class `wizard-card wizard-card--active`
   - Answered questions above: class `wizard-card wizard-card--answered`
   - Future questions below: class `wizard-card wizard-card--locked`

2. Each question card contains:
   - Question text in a paragraph
   - Yes/No button pair (only visible on active and answered cards)
   - For answered cards: show which answer was given (Yes highlighted, No highlighted)
   - For locked cards: question text only, grayed out, no buttons

3. Sub-question cards: When a parent question is answered Yes, render sub-question cards indented below it:
   - For exclusive sub-questions (DNS): radio-style buttons showing the sub-question text
   - For non-exclusive sub-questions (GCP Cloud Forwarding): toggle buttons for each option
   - Sub-question cards have class `wizard-card wizard-card--sub`

4. Progress indicator: A simple fraction text at the top (e.g., "Question 2 of 5") showing the count of answered + 1 (current) vs total top-level questions.

Determining active/answered/locked state:
- Walk questions in order. A question is "answered" if ALL its featureIds have a value in features, OR if it has subQuestions and either (a) the parent was answered No (no sub-features selected) or (b) at least one sub-feature is selected.
- The first non-answered question is "active".
- All questions after the active one are "locked".
- A standalone question (no sub-questions, single featureId) is answered if features[featureId] is true or the user explicitly said No. To track "No" answers, store the feature ID with value `false` in features (setFeature with enabled=false stores explicitly, vs. not-yet-answered which means the key doesn't exist at all). Update the `hasProviderData` function in state.js to count only `true` values: `Object.values(features).some(v => v === true)`.

When user clicks Yes on a standalone question: call `onAnswer(featureId, true)`.
When user clicks No on a standalone question: call `onAnswer(featureId, false)`.
When user clicks Yes on a parent question with sub-questions: reveal sub-question cards (don't select any feature yet — user must pick a sub-option).
When user clicks No on a parent question: call `onAnswer` with false for all sub-feature IDs.
When user selects a sub-option (exclusive): call `onAnswer(selectedId, true)` and `onAnswer(otherId, false)` for exclusive groups.
When user selects a sub-option (non-exclusive): toggle the specific featureId.

Update `updateWorkspace` to:
- When no provider selected: show workspace__empty, hide header + content
- When provider selected: hide workspace__empty, show header + content, set workspace title to provider display name + " Features"
