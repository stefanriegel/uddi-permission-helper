```
ec2:DescribeVpcs, ec2:DescribeSubnets, ec2:DescribeAddresses, ec2:DescribeRouteTables,
```

Terraform: `aws_iam_policy` for read-only (7 actions) or bidirectional (14 actions).
Setup guide: Create policy, name 'InfobloxUDDI-DnsRoute53', attach to role.

**5. `cloudForwarding` — has sub-features `discoveryOnly` and `fullManagement`:**

```
route53resolver:ListResolverEndpoints, route53resolver:ListResolverEndpointIpAddresses,
```

```
route53resolver:*,

ec2:DescribeAvailabilityZones, ec2:ModifyNetworkInterfaceAttribute,
ec2:CreateNetworkInterfacePermission, ec2:DescribeSecurityGroups
```
Note: `route53resolver:*` is a wildcard action — count it as 1 action in the array. Total: 12 actions.

Terraform: `aws_iam_policy` for discovery (6 actions) or full management (12 actions).
Setup guide: Create policy, name 'InfobloxUDDI-CloudForwarding', attach to role.

**6. `multiAccount`:**
This category is structurally different — it produces THREE separate policy documents:

a) **Trust policy template** for sub-account IAM roles:
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "AWS": "arn:aws:iam::902917483333:root" },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": { "sts:ExternalId": "<YOUR_EXTERNAL_ID>" }
    }
  }]
}
```

b) **Organizations read-only policy** (management account):
Reference `AWSOrganizationsReadOnlyAccess` managed policy ARN.

c) **STS AssumeRole policy** (management account IAM user):
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "sts:AssumeRole",
    "Resource": "arn:aws:iam::*:role/InfobloxUDDI-DiscoveryRole"
  }]
}
```

Terraform: `aws_iam_role` with assume_role_policy (trust policy), `aws_iam_role_policy_attachment` for Organizations, and `aws_iam_policy` for STS.
Setup guide: "1. In each sub-account: Create IAM role 'InfobloxUDDI-DiscoveryRole' with the trust policy. 2. Attach discovery policies to each sub-account role. 3. In management account: Attach AWSOrganizationsReadOnlyAccess. 4. Create STS policy allowing AssumeRole. 5. Enter External ID in Infoblox Portal."

**Data shape per feature:**
```javascript
{
  id: 'vpcIpamDiscovery',
  name: 'VPC/IPAM Discovery',
  question: 'Discover VPCs, subnets, IP address management?',
  actions: ['ec2:DescribeVpcs', ...],  // exact IAM action strings
  terraform: `resource "aws_iam_policy" ...`,  // complete HCL
  setupGuide: '1. Open IAM Console...'  // step-by-step text
}
```

For DNS and Cloud Forwarding sub-features, use separate entries (e.g., `dnsRoute53ReadOnly` and `dnsRoute53Bidirectional` as distinct feature objects) rather than nested sub-objects. This simplifies downstream consumption — the wizard (Phase 6) controls which sub-feature is selected.

For multiAccount, the shape differs: instead of `actions[]`, use `policies` array containing `{ name, type, document }` objects for the three policy documents.

Each feature should include a `rationale` object mapping each action to a short explanation string (e.g., `'ec2:DescribeVpcs': 'List VPCs for network topology discovery'`). This satisfies the requirement that each permission includes a rationale (OUT-07).
